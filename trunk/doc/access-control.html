Using Subversion properties for repository access control

Using properties in partnership with hooks and svnlook, we can provide a flexible, granular, visible and version controlled mechanism for access control independent of the repository access method.

Three properties are used. One to control who can grant commit rights, one to control who can commit, and one to control who can change revision properties. These properties can exist on individual files or directories, or on the parent directories of files or directories governed by access control. They do not have to exist in tandem in the same location, however all should exist in the repository at the top level of the path you wish to control.

Caveat: This approach does not provide user authentication. The repository access method will still need to provide authentication to the server.

property: hook:commit-access

This property contains a line delimited list of users who are permitted to commit the file or directory of the property in question. For directories, this permission cascades down to sub-directories and files within these sub-directories.

Conversely, this property can also explicitly deny access if any user is prefixed with an exclamation mark !. This can be useful when a user has been granted access rights higher up the repository path and you wish to deny at a lower path.

e.g. To grant user admin1 and user1 commit rights to any part of the repository, hook:commit-access is set to the following values on the repository root:

admin1
user1

property: hook:propchange-access

This property contains a line delimited list of properties. Each property is suffixed with an equals sign followed by a comma delimited list of users permitted to modify the property.

Conversely, this property can also explicitly deny access if any user is prefixed with an exclamation mark !.

To enable changes to access control, this property needs to set values for two properties, the first controls who is allowed to modify the hook:propchange-access property, the second controls who is allowed to modify the hook:commit-access property.

Excluding hook:propchange-access, hook:commit-access and hook:revpropchange-access properties, it is possible to add wildcard matches for properties using an asterix *.

e.g. To grant user admin1 rights to control who is allowed to modify the hook:propchange-access and hook:revpropchange-access properties, and users admin1 and user1 rights to modify the hook:commit-access property, and to grant user1 rights to modify any property, and apply all of these rights to any part of the repository, the hook:propchange-access property is set to the following values on the repository root:

hook:propchange-access=admin1
hook:revpropchange-access=admin1
hook:commit-access=admin1,user1
*=user1

If admin1 wanted to prevent user1 from modifying who can commit to a particular path in the repository, say in the /admin-only directory, then admin1 would set the following value to the hook:propchange-access property on the /admin-only directory:

hook:commit-access=!user1

If admin1 also wanted to prevent user1 from committing in /admin-only, then admin1 would set the following value to the hook:commit-access property on the /admin-only directory:

!user1

property: hook:revpropchange-access

This property contains a line delimited list of users permitted to change revision properties.

Conversely, this property can also explicitly deny access if any user is prefixed with an exclamation mark !.

e.g. To grant user admin1 and user1 rights to modify revision properties to any part of the repository, the hook:revpropchange-access property is set to the following values on the repository root:

admin1
user1

If admin1 also wanted to prevent user1 from committing in /admin-only, then admin1 would set the following value to the hook:revpropchange-access property on the /admin-only directory:

!user1

hook: pre-commit

This hook script searches HEAD revision for hook:commit-access properties on the files and directories being changed. If the property is not found, each parent directory back to the repository root is searched. If a matching property is found, the user's permission is verified and the search is stopped. If no match is found, permission is denied.

If a property is being modified, which can be determined by using the svnlook proplist command for the transaction being committed, the hook:propchange-access property is also searched on the HEAD revision on the files and directories changing. If the property is not found, each parent directory back to the repository root is searched until a match is found. If a matching property is found, the user's permission to modify the property is verified and the search is stopped. If no match is found, permission is denied.

Note that whilst the search for hook:commit-access and hook:propchange-access should both start on and be performed for each file or directory being modified (before working their way up the repository path), they are independant and should be treated separately. i.e. A match on hook:commit-access does not stop the search for hook:propchange-access properties, and vice versa.

hook: start-commit

This hook script searches HEAD revision for the hook:commit-access property on the repository root, and if found, verify user is permitted to commit. This enables basic repository access control for each user.

hook: pre-revprop-change

This hook script searches HEAD revision for hook:revpropchange-access properties on the files and directories being changed. If the property is not found, each parent directory back to the repository root is searched. If a matching property is found, the user's permission is verified and the search is stopped. If no match is found, permission is denied.
